# Сборщик телеметрии пользовательской активности (Windows)

Проект собирает телеметрию о действиях пользователя, состоянии системы и сетевом трафике.
Дополнительно может подхватывать последний CSV-лог имитатора и объединять все события в единый итоговый JSON-файл.

## Возможности

- Сбор пользовательской активности:
	- нажатия клавиш
	- клики мышью
	- смена активного окна и длительность фокуса
	- открытие и закрытие окон
	- перемещение и изменение размера окон
- Сбор системных метрик:
	- загрузка CPU
	- использование RAM
	- список процессов
	- использование дисков
- Сбор сетевой телеметрии через Scapy:
	- источник и назначение
	- протокол (IP/TCP/UDP/DNS)
	- длина пакета
- Импорт событий из последнего лога имитатора (`logs_dir/log_*.csv` - для этого telemetry.exe должен лежать в одной директории с `imitationagent.exe`)
- Формирование финального объединенного файла телеметрии с сортировкой по времени

## Требования

- ОС: Windows
- Python: 3.11+
- Права, достаточные для захвата сетевого трафика (администратор)
- Для Scapy в Windows обычно требуется установленный драйвер захвата (Npcap)

## Создание виртуального окружения

1. Создайте и активируйте виртуальное окружение в директории проекта:

```powershell
python -m venv venv
venv\Scripts\activate
```

2. Установите зависимости:

```powershell
pip install -r requirements.txt
```

## Быстрый старт

Запуск в режиме по умолчанию (с ожиданием имитатора, будет ждать пока имитатор не начнет работу):

```powershell
python main.py
```

Явно запуск с учетом имитатора:

```powershell
python main.py --with-imitator
```

Запуск независимо от имитатора:

```powershell
python main.py --no-imitator
```

## Режимы запуска

- `--with-imitator`:
	- если процесс `imitationagent.exe` не найден, приложение ждет его появления
	- пока имитатор работает, идет сбор телеметрии
	- когда имитатор завершился, сбор останавливается
	- затем подгружается последний CSV-лог имитатора и формируется итоговый файл
- `--no-imitator`:
	- сбор запускается сразу и работает до ручной остановки (Ctrl+C или через диспетчер задач)
	- при остановке также выполняется попытка подгрузить последний CSV-лог имитатора и собрать финальный файл

Если флаг не передан, используется поведение `--with-imitator`.

## Выходные данные

В каждом запуске создается отдельная директория телеметрии с файлами:

- `user_activity_*.json` - события пользовательской активности
- `system_metrics_*.json` - метрики CPU/RAM/дисков/процессов
- `network_metrics_*.json` - сетевые события
- `imitator_logs_*.json` - события, импортированные из CSV-лога имитатора
- `final_telemetry_*.json` - объединенный и отсортированный итог

Формат строк в JSON-файлах (JSON Lines):

```json
{
    "date":"2026-04-24",
    "time":"12:34:56.123456",
    "event_type":"user_activity",
    "data":{...}
}
```

## Формат лога имитатора

Для импорта ожидается CSV в директории `logs_dir` с именем:

```text
log_YYYY-MM-DD_HH-MM-SS.csv
```

Ожидаемый заголовок:

```csv
timestamp;level;message
```

Пример строк:

```csv
2026-04-23T00:41:24;INFO;Приложение запущено
2026-04-23T01:16:00;ERROR;файл module_custom_funcs.py: пример ошибки
```

## Сборка в EXE (Nuitka)

Сборка onefile-версии:

```powershell
python -m nuitka main.py --follow-imports --standalone --jobs=4 --onefile --disable-console --include-package=scapy --output-filename=telemetry.exe
```

Примечание: для диагностики рекомендуется оставить консоль (`--enable-console`).

## Важные замечания

- Проект использует низкоуровневые Windows API и глобальные слушатели клавиатуры/мыши.
- Сетевой захват может не работать без Npcap или при недостаточных правах.
- Если директория `logs_dir` отсутствует или в ней нет подходящих файлов, импорт логов имитатора будет пропущен с сообщением в логе.
